=======
Settings
=======

.. contents::
    :local:
    :depth: 1

.. warning::

    Be careful when you override settings, especially when the default value
    is a non-empty list or dictionary, such as :setting:`STATICFILES_FINDERS`.
    Make sure you keep the components required by the features of Django you
    wish to use.

Core Settings
=============

Here's a list of settings available in Django core and their default values.
Settings provided by contrib apps are listed below, followed by a topical index
of the core settings. For introductory material, see the :doc:`settings topic
guide </topics/settings>`.

.. setting:: ABSOLUTE_URL_OVERRIDES

``ABSOLUTE_URL_OVERRIDES``
--------------------------

Default: ``{}`` (Empty dictionary)

A dictionary mapping ``"app_label.model_name"`` strings to functions that take
a model object and return its URL. This is a way of inserting or overriding
``get_absolute_url()`` methods on a per-installation basis. Example::

    ABSOLUTE_URL_OVERRIDES = {
        'blogs.weblog': lambda o: "/blogs/%s/" % o.slug,
        'news.story': lambda o: "/stories/%s/%s/" % (o.pub_year, o.slug),
    }

The model name used in this setting should be all lowercase, regardless of the
case of the actual model class name.

.. setting:: ADMINS

``ADMINS``
----------

Default: ``[]`` (Empty list)

A list of all the people who get code error notifications. When
:setting:`DEBUG=False <DEBUG>` and :class:`~django.utils.log.AdminEmailHandler`
is configured in :setting:`LOGGING` (done by default), Django emails these
people the details of exceptions raised in the request/response cycle.

Each item in the list should be a tuple of (Full name, email address). Example::

    [('John', 'john@example.com'), ('Mary', 'mary@example.com')]

.. setting:: ALLOWED_HOSTS

``ALLOWED_HOSTS``
-----------------

Default: ``[]`` (Empty list)

A list of strings representing the host/domain names that this Django site can
serve. This is a security measure to prevent :ref:`HTTP Host header attacks
<host-headers-virtual-hosting>`, which are possible even under many
seemingly-safe web server configurations.

Values in this list can be fully qualified names (e.g. ``'www.example.com'``),
in which case they will be matched against the request's ``Host`` header
exactly (case-insensitive, not including port). A value beginning with a period
can be used as a subdomain wildcard: ``'.example.com'`` will match
``example.com``, ``www.example.com``, and any other subdomain of
``example.com``. A value of ``'*'`` will match anything; in this case you are
responsible to provide your own validation of the ``Host`` header (perhaps in a
middleware; if so this middleware must be listed first in
:setting:`MIDDLEWARE`).

Django also allows the `fully qualified domain name (FQDN)`_ of any entries.
Some browsers include a trailing dot in the ``Host`` header which Django
strips when performing host validation.

.. _`fully qualified domain name (FQDN)`: https://en.wikipedia.org/wiki/Fully_qualified_domain_name

If the ``Host`` header (or ``X-Forwarded-Host`` if
:setting:`USE_X_FORWARDED_HOST` is enabled) does not match any value in this
list, the :meth:`django.http.HttpRequest.get_host()` method will raise
:exc:`~django.core.exceptions.SuspiciousOperation`.

When :setting:`DEBUG` is ``True`` and ``ALLOWED_HOSTS`` is empty, the host
is validated against ``['localhost', '127.0.0.1', '[::1]']``.

``ALLOWED_HOSTS`` is also :ref:`checked when running tests
<topics-testing-advanced-multiple-hosts>`.

This validation only applies via :meth:`~django.http.HttpRequest.get_host()`
if your code accesses the ``Host`` header directly from ``request.META`` you
are bypassing this security protection.

.. setting:: APPEND_SLASH

``APPEND_SLASH``
----------------

Default: ``True``

When set to ``True``, if the request URL does not match any of the patterns
in the URLconf and it doesn't end in a slash, an HTTP redirect is issued to the
same URL with a slash appended. Note that the redirect may cause any data
submitted in a POST request to be lost.

The :setting:`APPEND_SLASH` setting is only used if
:class:`~django.middleware.common.CommonMiddleware` is installed
(see :doc:`/topics/http/middleware`). See also :setting:`PREPEND_WWW`.

.. setting:: CACHES

``CACHES``
----------

Default::

    {
        'default': {
            'BACKEND': 'django.core.cache.backends.locmem.LocMemCache',
        }
    }

A dictionary containing the settings for all caches to be used with
Django. It is a nested dictionary whose contents maps cache aliases
to a dictionary containing the options for an individual cache.

The :setting:`CACHES` setting must configure a ``default`` cache;
any number of additional caches may also be specified. If you
are using a cache backend other than the local memory cache, or
you need to define multiple caches, other options will be required.
The following cache options are available.

.. setting:: CACHES-BACKEND

``BACKEND``
~~~~~~~~~~~

Default: ``''`` (Empty string)

The cache backend to use. The built-in cache backends are:

* ``'django.core.cache.backends.db.DatabaseCache'``
* ``'django.core.cache.backends.dummy.DummyCache'``
* ``'django.core.cache.backends.filebased.FileBasedCache'``
* ``'django.core.cache.backends.locmem.LocMemCache'``
* ``'django.core.cache.backends.memcached.MemcachedCache'``
* ``'django.core.cache.backends.memcached.PyLibMCCache'``

You can use a cache backend that doesn't ship with Django by setting
:setting:`BACKEND <CACHES-BACKEND>` to a fully-qualified path of a cache
backend class (i.e. ``mypackage.backends.whatever.WhateverCache``).

.. setting:: CACHES-KEY_FUNCTION

``KEY_FUNCTION``
~~~~~~~~~~~~~~~~

A string containing a dotted path to a function (or any callable) that defines how to
compose a prefix, version and key into a final cache key. The default
implementation is equivalent to the function::

    def make_key(key, key_prefix, version):
        return ':'.join([key_prefix, str(version), key])

You may use any key function you want, as long as it has the same
argument signature.

See the :ref:`cache documentation <cache_key_transformation>` for more
information.

.. setting:: CACHES-KEY_PREFIX

``KEY_PREFIX``
~~~~~~~~~~~~~~

Default: ``''`` (Empty string)

A string that will be automatically included (prepended by default) to
all cache keys used by the Django server.

See the :ref:`cache documentation <cache_key_prefixing>` for more information.

.. setting:: CACHES-LOCATION

``LOCATION``
~~~~~~~~~~~~

Default: ``''`` (Empty string)

The location of the cache to use. This might be the directory for a
file system cache, a host and port for a memcache server, or simply an
identifying name for a local memory cache. e.g.::

    CACHES = {
        'default': {
            'BACKEND': 'django.core.cache.backends.filebased.FileBasedCache',
            'LOCATION': '/var/tmp/django_cache',
        }
    }

.. setting:: CACHES-OPTIONS

``OPTIONS``
~~~~~~~~~~~

Default: ``None``

Extra parameters to pass to the cache backend. Available parameters depend on
the cache backend you're using.

.. setting:: CACHES-TIMEOUT

``TIMEOUT``
~~~~~~~~~~~

Default: 5 minutes (300 seconds)

Default timeout, in seconds, for the cache. If this argument is ``None``, entries
will cache forever.

.. setting:: CACHES-VERSION

``VERSION``
~~~~~~~~~~~

Default: ``1``

The default version number for cache keys generated by Django.

.. setting:: CACHE_MIDDLEWARE_ALIAS

``CACHE_MIDDLEWARE_ALIAS``
--------------------------

Default: ``'default'``

The cache connection to use for :class:`~django.middleware.cache.FetchFromCacheMiddleware`
and :class:`~django.middleware.cache.UpdateCacheMiddleware`.

.. setting:: CACHE_MIDDLEWARE_KEY_PREFIX

``CACHE_MIDDLEWARE_KEY_PREFIX``
-------------------------------

Default: ``''`` (Empty string)

A prefix for the cache keys that are used by the :class:`~django.middleware.cache.UpdateCacheMiddleware`
and :class:`~django.middleware.cache.FetchFromCacheMiddleware` middleware.

See :ref:`the caching documentation <cache_key_prefixing>`.

.. setting:: CACHE_MIDDLEWARE_SECONDS

``CACHE_MIDDLEWARE_SECONDS``
----------------------------

Default: ``600``

Default number of seconds to cache a page when using the cache middleware.

.. setting:: CSRF_COOKIE_AGE

``CSRF_COOKIE_AGE``
-------------------

Default: ``31449600`` (one year, in seconds)

The age of the CSRF cookie, in seconds.

.. setting:: CSRF_COOKIE_DOMAIN

``CSRF_COOKIE_DOMAIN``
----------------------

Default: ``None``

The domain to be used when setting the CSRF cookie. This can be useful to
allow cross-subdomain requests to be excluded from the normal cross site
request forgery protection. It should be set to a value such as
``".example.com"`` to allow a POST request from a subdomain of
``example.com`` to be processed.

Please note that the presence of this setting does not imply that CSRF
protection is disabled outside of the specified domain(s). The CSRF
protection is still active for all requests that originate from origins
outside the cookie domain.

.. setting:: CSRF_COOKIE_HTTPONLY

``CSRF_COOKIE_HTTPONLY``
------------------------

Default: ``False``

Whether to use ``HttpOnly`` flag on the CSRF cookie. If this is set to
``True``, client-side JavaScript will not be able to access the CSRF cookie.

.. setting:: CSRF_COOKIE_NAME

``CSRF_COOKIE_NAME``
--------------------

Default: ``'csrftoken'``

The name of the cookie to use for the CSRF authentication token. This can be
whatever you want (as long as it's different from the other cookie names your
application uses). See :ref:`how-csrf-works` for more information.

.. setting:: CSRF_COOKIE_PATH

``CSRF_COOKIE_PATH``
--------------------

Default: ``'/'``

The path set on the CSRF cookie. This should either match the path of your
Django installation or be a parent of that path.

For example, if you have your Django admin at ``/admin/``, the default
``CSRF_COOKIE_PATH`` of ``'/'`` will work. If you restricted the admin to
``/admin/mycustompath/``, you'd need to set this setting to
``'/admin/mycustompath/'`` for the CSRF protection to work as expected in the
admin views.

.. setting:: CSRF_COOKIE_SECURE

``CSRF_COOKIE_SECURE``
----------------------

Default: ``False``

Whether to use a secure cookie for the CSRF cookie when issuing the cookie.
If this is set to ``True``, the cookie will only be sent by the browser to an
HTTPS connection.

.. setting:: CSRF_COOKIE_SAMESITE

``CSRF_COOKIE_SAMESITE``
------------------------

Default: ``'Lax'``

The value of the ``SameSite`` attribute of the CSRF cookie when issuing the
cookie. This attribute set to ``'Strict'`` or ``'Lax'`` prevent the CSRF
cookie from being sent cross-site.

.. setting:: CSRF_FAILURE_VIEW

``CSRF_FAILURE_VIEW``
---------------------

Default: ``'django.views.csrf.csrf_failure'``

A dotted path to the view function to be used when an incoming request is
rejected by the CSRF middleware. The function should have this signature::

    def csrf_failure(request, reason=""):
        ...

See :ref:`the CSRF documentation <csrf_failure_view>` for more information.

.. setting:: CSRF_HEADER_NAME

``CSRF_HEADER_NAME``
--------------------

Default: ``'HTTP_X_CSRFTOKEN'``

The name of the request header Django uses for the CSRF authentication
token. Note that the name has Django's standard HTTP header conversion:
all underscores are converted to upper case, and the ``HTTP_`` prefix is
added.

.. setting:: CSRF_TRUSTED_ORIGINS

``CSRF_TRUSTED_ORIGINS``
------------------------

Default: ``[]`` (Empty list)

A list of trusted origins that are exempt from the CSRF protection. Each entry
can contain a protocol, domain and optional port, as listed below:

* ``'https://example.com'`` - will match both the domain and subdomain
* ``'https://*.example.com'`` - subdomain wildcard with explicit scheme
* ``'example.com'`` - will match all schemes

Subdomains of an entry are considered trusted when the setting is a bare
domain. If you're enabling CSRF protection for cross-domain requests (as opposed
to disabling it), you should set this to a more restrictive list. For example::

    CSRF_TRUSTED_ORIGINS = [
        'https://example.com',
        'https://api.example.com',
    ]

An asterisk is allowed and is the equivalent of all schemes but matching is
at the domain level, not the scheme level. For example::

    CSRF_TRUSTED_ORIGINS = [
        '*.example.com',  # Matches all schemes and all subdomains of example.com
    ]

.. setting:: CSRF_USE_SESSIONS

``CSRF_USE_SESSIONS``
---------------------

Default: ``False``

Whether to store the CSRF token in the user's session instead of in a cookie.
This requires :class:`~django.contrib.sessions.middleware.SessionMiddleware`
to be enabled. See :ref:`how-csrf-works` for more information.

.. setting:: DATABASES

``DATABASES``
-------------

Default: ``{}`` (Empty dictionary)

A dictionary containing the settings for all databases to be used with Django.
It is a nested dictionary whose contents map a database alias to a dictionary
containing the options for an individual database.

The :setting:`DATABASES` setting must configure a ``default`` database;
any number of additional databases may also be specified.

The simplest possible settings file is one that uses a SQLite database::

    DATABASES = {
        'default': {
            'ENGINE': 'django.db.backends.sqlite3',
            'NAME': 'mydatabase',
        }
    }

When connecting to other database backends, such as MySQL, Oracle, or
PostgreSQL, additional connection parameters will be required. See the
:setting:`ENGINE` documentation for details. For that purpose, the following
inner keys may be defined:

.. setting:: ENGINE

``ENGINE``
~~~~~~~~~~

Default: ``''`` (Empty string)

The database backend to use. The built-in database backends are:

* ``'django.db.backends.postgresql'``
* ``'django.db.backends.mysql'``
* ``'django.db.backends.sqlite3'``
* ``'django.db.backends.oracle'``

You can use a database backend that doesn't ship with Django by setting
:setting:`ENGINE` to a fully-qualified path (i.e.
``mypackage.backends.whatever``).

.. setting:: NAME

``NAME``
~~~~~~~~

Default: ``''`` (Empty string)

The name of the database to connect to. This is required for most database backends.
If you're using SQLite, specify the full filesystem path to the database file.
When specifying the path, always use forward slashes, even on Windows
(e.g. ``C:/homes/user/mysite/db.sqlite3``).

.. setting:: USER

``USER``
~~~~~~~~

Default: ``''`` (Empty string)

The username to use when connecting to the database. Not used with SQLite.

.. setting:: PASSWORD

``PASSWORD``
~~~~~~~~~~~~

Default: ``''`` (Empty string)

The password to use when connecting to the database. Not used with SQLite.

.. setting:: HOST

``HOST``
~~~~~~~~

Default: ``''`` (Empty string)

The host to use when connecting to the database. An empty string means
localhost. Not used with SQLite.

If this value contains a colon (i.e. ``hostname:port``) as with a PostgreSQL
default of ``localhost:5432``, the colon will cause an error. Always specify
the host and port separately (see :setting:`PORT` below).

.. setting:: PORT

``PORT``
~~~~~~~~

Default: ``''`` (Empty string)

The port to use when connecting to the database. An empty string means the
default port. Not used with SQLite.

.. setting:: ATOMIC_REQUESTS

``ATOMIC_REQUESTS``
~~~~~~~~~~~~~~~~~~~

Default: ``False``

If this is set to ``True``, Django wraps the execution of each view in a
database transaction. See :doc:`/topics/db/transactions`.

.. setting:: AUTOCOMMIT

``AUTOCOMMIT``
~~~~~~~~~~~~~~

Default: ``True``

If this is set to ``False``, Django will disable the automatic transaction
commit. Database operations need to be committed explicitly using ``transaction.commit()``.

This is useful when using a database that doesn't support automatic rollback or
for performing custom transaction handling within views or request handlers.

.. setting:: CONN_MAX_AGE

``CONN_MAX_AGE``
~~~~~~~~~~~~~~~~

Default: ``0``

The lifetime of a database connection, in seconds. Use ``0`` to close
database connections at the end of each request and ``None`` for unlimited
persistent connections. The default value of ``0`` is recommended for
development.

.. setting:: OPTIONS

``OPTIONS``
~~~~~~~~~~~

Default: ``{}`` (Empty dictionary)

Extra parameters to pass to the database backend. Available parameters
depend on your database backend.

.. setting:: TIME_ZONE

``TIME_ZONE``
-------------

Default: ``'America/Chicago'``

A string representing the time zone for this installation. This should be
a string such as ``'America/Chicago'`` or ``'Europe/London'``. This is used
when Django needs to display a time, and needs to convert from UTC to a local
time. When ``USE_TZ`` is ``True``, this is the timezone that Django will
use to display datetimes in templates and to interpret datetimes entered
in forms.

See the `list of valid time zone strings
<https://en.wikipedia.org/wiki/List_of_tz_zones_by_name>`_.

.. setting:: USE_TZ

``USE_TZ``
----------

Default: ``False``

A boolean that specifies whether Django should use timezone-aware datetimes
internally. If ``True``, Django will store datetimes in UTC in the database,
use timezone-aware datetimes internally, and return timezone-aware datetimes
when reading data.

If ``False``, Django will use naive datetimes (without timezone information)
throughout the application.

See the :doc:`time zone documentation </topics/i18n/timezones>` for more
information on timezone handling.

.. setting:: LANGUAGE_CODE

``LANGUAGE_CODE``
-----------------

Default: ``'en-us'``

A string representing the language code for this installation. This should be
a string in the format ``xx-yy`` or ``xx``, where ``xx`` is the language code
and ``yy`` is the optional country code. Examples are ``'en'``, ``'en-us'``,
``'fr'``, etc.

.. setting:: LANGUAGES

``LANGUAGES``
-------------

Default: Globally available languages

A list of all available languages. Each item in this list is a two-tuple of a
language code and a human-readable name.

.. setting:: LOCALE_PATHS

``LOCALE_PATHS``
----------------

Default: ``[]`` (Empty list)

A list of directories where Django looks for translation files. See
:doc:`/topics/i18n/translation`.

.. setting:: USE_I18N

``USE_I18N``
------------

Default: ``True``

A boolean that specifies whether Django's internationalization system should be
enabled. This provides an easy way to turn it off, for performance reasons.
If this is set to ``False``, Django will make some optimizations so as not to
load the internationalization machinery.

.. setting:: USE_L10N

``USE_L10N``
------------

Default: ``False``

A boolean that specifies whether localized formatting of data should be enabled.
When ``True``, Django will display numbers and dates using the format of the
current locale.

.. setting:: USE_THOUSAND_SEPARATOR

``USE_THOUSAND_SEPARATOR``
--------------------------

Default: ``False``

A boolean that specifies whether to add thousand separators when displaying
large numbers using Django templates.

.. setting:: ADMINS

``ADMINS``
----------

Default: ``[]`` (Empty list)

A list of all the people who get code error notifications. When
:setting:`DEBUG=False <DEBUG>` and :setting:`LOGGING` is configured with
an :class:`~django.utils.log.AdminEmailHandler`, Django emails these
people the details of exceptions raised in the request/response cycle.

Each item in the list should be a tuple of (Full name, email address).
Example::

    [('John', 'john@example.com'), ('Mary', 'mary@example.com')]

.. setting:: MANAGERS

``MANAGERS``
------------

Default: ``[]`` (Empty list)

A list of people who should get broken link notifications. When
``BrokenLinkEmailsMiddleware`` is enabled and
:setting:`DEBUG=False`, Django emails these people about broken links on your site.

Each item in the list should be a tuple of (Full name, email address).

.. setting:: AUTH_USER_MODEL

``AUTH_USER_MODEL``
-------------------

Default: ``'auth.User'``

The model to use to represent a User. If you substitute a custom user model,
this setting should be the full Python import path of that model. See
:ref:`auth-custom-user`.

.. setting:: FILE_UPLOAD_MAX_MEMORY_SIZE

``FILE_UPLOAD_MAX_MEMORY_SIZE``
-------------------------------

Default: 2.5MB

The maximum size (in bytes) that an upload will be before it gets streamed to
the file system. See :ref:`file-upload-security-considerations` for details.

.. setting:: FILE_UPLOAD_TEMP_DIR

``FILE_UPLOAD_TEMP_DIR``
------------------------

Default: ``None``

The directory to use for temporary file storage when a user uploads a file
larger than ``FILE_UPLOAD_MAX_MEMORY_SIZE``. ``None`` means Django will use
the system's temporary directory.

.. setting:: FILE_UPLOAD_PERMISSIONS

``FILE_UPLOAD_PERMISSIONS``
---------------------------

Default: ``0o644``

The numeric mode (i.e. ``0o644``) to set newly uploaded files to. For
more information about what these modes mean, see the documentation for
:func:`os.chmod`.

If this is set to ``None``, you'll get operating-system
dependent behavior. On most platforms, temporary files will have a mode
of ``0o600``, and files saved from memory will be saved using the
system's standard umask.

For security reasons, these permissions aren't applied to the temporary files
that are stored in :setting:`FILE_UPLOAD_TEMP_DIR`.

This setting also determines the default permissions for collected static files
when using the :djadmin:`collectstatic` management command. See
:djadmin:`collectstatic` for details on overriding it.

.. warning::

    **Always prefix the mode with a 0.**

    If you're not familiar with file modes, please note that the leading
    ``0`` is very important: it indicates an octal number, which is the
    notation used for Unix permissions. If you try to use ``644``, you'll
    get totally incorrect behavior.

.. setting:: FILE_UPLOAD_DIRECTORY_PERMISSIONS

``FILE_UPLOAD_DIRECTORY_PERMISSIONS``
-------------------------------------

Default: ``None``

The numeric mode (i.e. ``0o755``) to set newly created directories to.
See :setting:`FILE_UPLOAD_PERMISSIONS` for more details. In the case of 
``None``, the operating system uses its default.

.. warning::

    **Always prefix the mode with a 0.**

    If you're not familiar with file modes, please note that the leading
    ``0`` is very important: it indicates an octal number, which is the
    notation used for Unix permissions.

.. setting:: DATA_UPLOAD_MAX_MEMORY_SIZE

``DATA_UPLOAD_MAX_MEMORY_SIZE``
-------------------------------

Default: 2.5MB

The maximum size (in bytes) that a request body may be before a
:exc:`~django.core.exceptions.SuspiciousOperation` (RequestDataTooBig) is raised.
The check is done when accessing ``request.body`` or ``request.POST`` and is
calculated against the total request size excluding any file upload data. You
can set this to ``None`` to disable the check. Applications that are expected
to receive unusually large form submissions should tune this setting.

The amount of data requested in a single request from a client is usually
specified by the DATA_UPLOAD_MAX_NUMBER_FIELDS setting.

.. setting:: DATA_UPLOAD_MAX_NUMBER_FIELDS

``DATA_UPLOAD_MAX_NUMBER_FIELDS``
---------------------------------

Default: 1000

The maximum number of fields that will be accepted before a
:exc:`~django.core.exceptions.SuspiciousOperation` (TooManyFieldsSent) is raised.
This is an unrelated setting to the other ``DATA_UPLOAD_MAX_*`` settings.
If you encounter this error, it most likely means you have a form that exceeds
this number. You can increase the limit by updating this setting. Note that the
limit is per request/form, so multiple forms on the same page each count separately.

For backwards compatibility, this is set to ``1000``.

.. setting:: DATE_FORMAT

``DATE_FORMAT``
---------------

Default: ``'N j, Y'`` (e.g. ``Feb. 4, 2003``)

The default formatting to use for displaying dates in Django templates. Note
that if :setting:`USE_L10N` is set to ``True``, the corresponding
locale-dictated format has higher precedence and will be applied.

.. setting:: DATE_INPUT_FORMATS

``DATE_INPUT_FORMATS``
----------------------

Default: ``['%Y-%m-%d', '%m/%d/%Y', ...]`` (see implementation)

A list of formats that will be accepted when inputting data on a date field.
Formats will be tried in order, using the first valid parsing. The default
value is used in a Django form field to parse the data. The format in
:setting:`DATE_FORMAT` is not used for parsing.

See the :doc:`date input format documentation </ref/forms/fields#date>`
for more details. For information about date/time input validation, see
:doc:`Form validation </ref/forms/validation>`.

.. setting:: DATE_INPUT_FORMATS

``DATETIME_FORMAT``
-------------------

Default: ``'N j, Y, P'`` (e.g. ``Feb. 4, 2003, 4 p.m.``)

The default formatting to use for displaying datetime objects in Django
templates. Note that if :setting:`USE_L10N` is set to ``True``, the
corresponding locale-dictated format has higher precedence and will be applied.

.. setting:: DATETIME_INPUT_FORMATS

``DATETIME_INPUT_FORMATS``
--------------------------

Default: ``['%Y-%m-%d %H:%M:%S', '%Y-%m-%d %H:%M', ...]`` (see implementation)

A list of formats that will be accepted when inputting data on a datetime
field. Formats will be tried in order, using the first valid parsing.

.. setting:: DEBUG

``DEBUG``
---------

Default: ``False``

A boolean that turns on/off debug mode. When set to ``True``, Django will
display detailed error pages with a lot of debugging information when an
unhandled exception is raised. These error pages contain sensitive information
about your project, SQL statements, and settings, and should never be enabled
on a public site.

Always set this to ``False`` on a production site. See the
:doc:`Security checklist </topics/security>` for more details.

.. setting:: DEFAULT_FROM_EMAIL

``DEFAULT_FROM_EMAIL``
----------------------

Default: ``'webmaster@localhost'``

Default email address to use for various automated correspondence from the
site managers.

.. setting:: DISALLOWED_USER_AGENTS

``DISALLOWED_USER_AGENTS``
--------------------------

Default: ``[]`` (Empty list)

List of compiled regular expression objects representing User-Agent strings
that are not allowed to visit any page, systemwide. Use this for bad
robots/crawlers. Here are a few examples::

    import re
    DISALLOWED_USER_AGENTS = [
        re.compile(r'^NaverBot.*'),
        re.compile(r'^EmailSiphon.*'),
    ]

Notice the raw string syntax (r'...'). Use this whenever specifying a regex.

.. setting:: EMAIL_BACKEND

``EMAIL_BACKEND``
-----------------

Default: ``'django.core.mail.backends.smtp.EmailBackend'``

The backend to use for sending emails. For the list of available backends
see :doc:`/topics/email`.

.. setting:: EMAIL_FILE_PATH

``EMAIL_FILE_PATH``
-------------------

Default: Not set

The directory used by the file email backend to store output files (when using
``'django.core.mail.backends.filebased.EmailBackend'``).

.. setting:: EMAIL_HOST

``EMAIL_HOST``
--------------

Default: ``'localhost'``

The host to use for sending email via SMTP (when using the default SMTP backend).

.. setting:: EMAIL_HOST_PASSWORD

``EMAIL_HOST_PASSWORD``
-----------------------

Default: ``''`` (Empty string)

Password to use for the SMTP server defined in :setting:`EMAIL_HOST`. This
setting is used in conjunction with :setting:`EMAIL_HOST_USER` when
authenticating to the SMTP server.

.. setting:: EMAIL_HOST_USER

``EMAIL_HOST_USER``

.. setting:: SECURE_PROXY_SSL_HEADER

``SECURE_PROXY_SSL_HEADER``
---------------------------

Default: ``None``

A tuple representing a HTTP header/value combination that signifies a request
is secure. This only affects the request if ``DEBUG`` is ``False`` and a
``Vary: Accept-Encoding`` header is present.

In a typical setup where Django is behind a reverse-proxy configured for
``HTTPS``, the proxy will set a header (e.g. ``X-Forwarded-Proto``) signifying
the original request was ``HTTPS``. Django should trust this header if and only
if the proxy reliably sets this header and you can trust the proxy's network.

This setting should be a tuple of ``(header_name, header_value)``. For example::

    SECURE_PROXY_SSL_HEADER = ('HTTP_X_FORWARDED_PROTO', 'https')

This means that if the ``X-Forwarded-Proto`` header is set to ``https``, then
Django considers the request to be over ``HTTPS``.

.. note::

    This setting should only be used when you control the proxy that sets this
    header. If you receive arbitrary values in this header from a client, you
    open yourself up to a security vulnerability. Django will trust that this
    header signifies a secure connection, so a client could exploit this by
    spoofing the header.

.. warning::

    If ``SECURE_PROXY_SSL_HEADER`` is enabled, ensure that your proxy only
    sends this header when appropriate. Otherwise, the header can be spoofed
    to claim a request is secure when it isn't, exposing your site to
    potential attacks.

Default: ``.`` (a period/full stop)

The decimal separator symbol to be used when formatting decimal numbers.

Contrib app settings
====================

When you use :doc:`django.contrib </ref/contrib/index>` apps in your project,
the settings that control their behavior are documented in that app's documentation
alongside the app's functionality. The settings in this document apply to
Django's core functionality and the apps in the ``django.contrib`` package.

Some settings that were previously documented here are deprecated and have been
removed.

To look up settings for third-party Django apps, see:

* :doc:`django-extensions </ref/contrib/commands>` - Additional Django management commands
* :doc:`django-filter </ref/apps>` - Reusable application for filtering
* :doc:`Django REST Framework </api-guide/settings>` - Configuration for Django REST
